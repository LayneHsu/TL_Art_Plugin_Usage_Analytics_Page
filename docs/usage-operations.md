# Usage operations

## Replay

`FirestoreReplayService` rebuilds a versioned shadow generation from raw events ordered by `(corrected_observed_at, server_received_at, event_id)` inside a company-time day-aligned half-open `[from, to)` window. A run explicitly declares `partition` or `global`; retained raw-event coverage is never used to guess the scope. `usageReplayJobs` persists the tuple checkpoint, catch-up fence, date revision vector, validation stage/cursor/totals, and cutover state. `usageReplayLocks` provides an expiring single-worker lease; `usageReplayGenerations` prevents generation reuse; `usageReplayAppliedEvents` makes page retries idempotent; `usageReplayValidationGroups` stores bounded expected groups; and `usageAggregateSourceRevisions` is updated in the online ingestion transaction for each corrected company date.

`ReplayService` is an in-memory fixture helper for unit/emulator comparisons only. It is not a production replay implementation and does not define range, watermark, concurrency, validation, rollback, or retention behavior. Production replay approval and execution wiring must instantiate `FirestoreReplayService`; deploying an endpoint or scheduled job backed by `ReplayService` is prohibited.

`usageReplayAdmin` is an IAM-private HTTP entry and accepts only the allowlisted Google service-account OIDC policy already used by plugin operations. A requester stores an immutable run, rollback, or finalize payload in `usageReplayApprovals`; a different service-account actor approves it; execution revalidates the exact payload and transactionally claims the review once. `finalize` is required after the rollback verification window: it keeps the selected reader generation, removes obsolete rollback routing, and stops indefinite dual writes. Portal Firebase roles, plugin leases, caller-provided worker IDs, and modified execution payloads are rejected. Large rebuilds may require a new approved execution to resume the persisted replay checkpoint after a bounded invocation.

Validation reads source and shadow group documents in bounded pages, persists a cursor and chained digest after every page, and compares complete daily, principal, fingerprint, identity-field, duration, and safe-summary data. Cutover transactionally rechecks the captured per-date revisions, so a late in-window commit after validation cannot be hidden by the pointer switch. Writes outside a partial replay window do not advance its validation revision. A replay may replace an exact existing partition but cannot partially overlap one; rollback restores the replaced partition generation and finalize clears its rollback generation. An explicitly global replay is the compaction path that replaces all date partitions after validating its complete declared range.

## Retention

`RetentionCleanupService` requires an explicit policy. It refuses to start if raw event retention is shorter than `rebuildWindow + lateArrivalAllowance`. Cleanup is bounded to Firestore-safe batches, supports dry-run, and writes an audit record for every collection batch before deleting anything. Raw-event deletion reads active replay jobs and the shared replay maintenance lock in the same transaction that deletes the selected page; a replay that starts after page listing therefore defers deletion before any source event is removed. Production policy values remain an owner decision in `docs/data-retention.md`.

The unqualified `RetentionCleanupService` is an in-memory adapter helper for unit tests and is not exported from the Functions package. Production scheduling must use `FirestoreRetentionCleanupService`, whose delete transactions enforce the retention lease and replay lock. Wiring the generic adapter helper to `FirestoreRetentionAdapter` is prohibited.

`retentionCleanupScheduled` reads the parameterized schedule, time zone, dry-run flag, and policy JSON. Its run ID is derived from the scheduler job and schedule time, so retries resolve the same candidate. `usageRetentionSchedules` keeps one Admin SDK-only active-run pointer per scheduler job: a later trigger resumes an unfinished run, and only a completed run permits a new schedule cycle. A short reservation covers the transaction gap before the run document is created. Each invocation uses a unique lease owner. `usageRetentionRuns` persists the stable page cursor, first-run cutoff, policy digest, and totals; completed historical runs are cleaned with `retentionRunRetentionMs`, while the current running run is excluded. Only `failed` and `finalized` `usageReplayJobs` are cleaned with `replayMetadataRetentionMs`; `running`, `switched`, and `rolled_back` jobs are excluded because rollback context remains active until finalize. Both query selection and deletion transactions enforce these states. The worker renews its lease around every page and rechecks ownership in deletion transactions. Lost ownership stops the run before deletion or checkpoint writes. `usageRetentionAudit` stores ID-hash-only page audits before deletion. Missing or changed policy fails closed.

Replay applied-event markers, generation claims, validation groups, replay locks, aggregate pointers, retention schedule pointers, retention audit records, and monitoring alert state are not deleted by this generic job. Their lifecycle is coupled to replay completion, rollback finalization, generation non-reuse, scheduler retirement, or bounded-state verification. Production must have a separate replay-aware purge/legal-hold owner and age/volume alert before ingestion is enabled; otherwise this is a release blocker.

## Monitoring

`evaluateUsageMonitoring` evaluates aggregate drift, permanent rejection rate, plugin authentication failures, lease renewal failures, dead-letter growth, and writes per accepted event. Every alert includes the configured owner and is warning/critical based on threshold multiples. Alert routing and on-call ownership must be supplied by deployment configuration; no portal role grants access to plugin operations.

`usageMonitoringScheduled` reads its parameterized schedule, time zone, thresholds, owner, and named routes. Aggregate drift compares the complete persisted replay source/shadow validation result, including group digests, rather than relying only on event counts or mirrored ingestion counters. `usageMonitoringCounters` stores independent operational counters, `usageMonitoringSnapshots` records evaluated metric/threshold snapshots, `usageMonitoringAlerts` keeps one deduplicated active/recovered document per alert code, and `usageMonitoringNotifications` is the bounded Admin SDK-only outbox consumed by configured route workers. Portal clients cannot read or write operational monitoring collections.

## Failure handling

Lease expiry is an authentication failure. A response that reports `renewal_recommended` should renew before retrying. Revocation and principal disable are immediate server decisions. Rate limits and transient writes are retryable with a bounded delay; invalid schema, registry versions, tool keys, identities, hard-range client clocks, or sensitive event data are permanent quarantine decisions and must not loop forever.
