# Plugin authentication contract

Plugin authentication is a separate security domain from portal Firebase authentication. A portal Firebase ID token, portal UID, portal role, or portal session is never accepted by a plugin endpoint. The browser pairing route does not initialize Firebase Auth and uses only `plugin_pairing_*` sessionStorage keys for the current one-time claim.

## One-time pairing

The plugin creates a random device ID, a random device challenge, and a short-lived pairing request. The browser receives the pairing ID in the query string and the one-time pairing secret only in the URL fragment (`#pairing_secret=...`). The page scrubs the fragment immediately before any OAuth or API work. It then explicitly starts a company Google OAuth/OIDC authorization-code flow with PKCE. The backend binds the request to the exact callback URI, OAuth state, OIDC nonce, PKCE challenge, device ID digest, device challenge digest, five-minute expiry, and the plugin client version.

The backend exchanges the code and verifies its signature, plugin-specific audience, allowed issuer, nonce, `email_verified=true`, and the configured company domain. It derives `plugin_principal_id` from a domain-separated keyed digest of immutable issuer + subject. Email, name, and avatar are display snapshots only. Identity failures use a generic response and cannot reveal whether an email, principal, or other binding exists.

A pairing is consumed at most once and creates at most one binding. Browser completion marks it authorized, but the long-term device credential is not created until the initiating plugin polls with the original device challenge. The successful poll atomically consumes the pairing, creates the binding, and returns a credential derived from a server-only delivery key, binding ID, and random delivery nonce. Only the credential digest and nonsecret delivery metadata are stored. If the HTTP response is lost, the same device proof and pairing secret can deterministically replay the same credential for 120 seconds. An explicit delivery acknowledgement is idempotent and clears replay capability; replay after acknowledgement, expiry, wrong proof, or wrong binding fails. Cancel, verification failure, callback/state/PKCE mismatch, timeout, or acknowledged consumption makes the pairing unusable. Browser pagehide and terminal errors submit best-effort cancellation, while intentional OAuth navigation is exempt.

Every successful `poll-pairing` response, including a response-loss replay, also contains the server-derived profile snapshot `email`, `display_name`, `avatar_url`, and `profile_updated_at`. The same four fields are included in every successful `renew-lease` response. They are read from the current enabled `PluginPrincipalRecord` after the device proof or credential is authorized; request-body profile fields and Windows user names are never consulted. A disabled principal or revoked binding receives the generic denial and no profile. Missing profile values are `null`; `profile_updated_at` is a stable ISO-8601 UTC string when present, and `avatar_url` is returned only when the stored backend-verified value is an HTTPS URL within the server length bound. The profile snapshot never includes issuer, subject, Firebase UID, any credential or secret field, raw principal data, or portal roles. The poll protocol's existing one-time `deviceCredential` delivery remains separate from this profile allowlist. Replays therefore return the latest enabled profile snapshot rather than a stale value.

## Device credential and rotation

A device credential authorizes only lease exchange and credential lifecycle endpoints. It does not authorize event writes or portal reads. A binding records its principal, device digest, client version, creation time, last verification time, credential version, and revocation state.

Rotation has explicit `prepare -> confirm`, `prepare -> cancel`, and `prepare -> expiry` transitions. Prepare returns the new random credential once and stores only its digest. The old credential remains current until the client confirms that the new credential was saved in secure storage by presenting it to confirm. Cancel, secure-store failure, or expiry leaves the old credential valid. Confirmation atomically installs the new digest and invalidates the old credential. The server retains only a bounded hash marker for the last confirmed rotation, so an identical retry after a lost response succeeds idempotently; a different credential or rotation ID is rejected.

## Lease and event decision

Lease renewal checks the current binding digest, binding revocation, and the plugin principal's independent enabled state. The service controls issuer, audience, version, key ID, JTI, principal, binding, issued time, and expiry. The maximum lifetime is 3600 seconds. Verification accepts an explicitly configured overlap set of signing key IDs so key rotation does not require every plugin to upgrade at once.

Initial delivery credentials are versioned by a stored delivery key ID. The server keeps the previous delivery key in a bounded overlap set for the 120-second response-loss replay and acknowledgement window, and derives with that original key rather than whichever key is currently active. An unavailable key or digest mismatch revokes the pending binding atomically and is audited; a delivery secret must never be replaced in place under the same ID.

The lease belongs in process memory only. A plugin process must not persist it, and a restarted process must renew online before enabling tools; an old in-memory lease cannot be reused offline after restart. Short network loss can continue only until the current server expiry.

Task 4 event authorization must call `PluginAuthDecisionService.authorizeEvent`. That decision verifies the signature and time, requires the queue `binding_id` to equal the lease binding, and rechecks the current binding and principal records. Revocation therefore blocks event upload immediately even if a previously issued lease has time remaining.

Offline queues are permanently bound to their original `binding_id`. Unlink, device revocation, or account switch cannot relabel a queue or upload it with another binding's lease. The client records only isolation/cleanup status such as retained, quarantined, or expired; event content is not stored in the authentication store. Historical accepted events remain attributed to their original immutable principal.

## Audit boundary

`pluginAuthAudit` records pairing lifecycle, binding creation, lease issuance, rotation transitions, unlink, device revocation, principal disable, signing-key rotation metadata, operations review request/approval/execute, and denied authorization. Event authorization denials include fixed reason codes for invalid signature, expiry, cross-binding queue, missing/revoked binding, and disabled/missing principal. Audit top-level identifiers, action, outcome, and bounded details are validated and sanitized; invalid or secret-bearing values become null or a generic denial and are never persisted. Records never contain authorization codes, ID tokens, portal tokens, Google ops bearer tokens, device credentials, PKCE verifiers, pairing secrets, or event content.

Event authorization remains fail closed if its denial audit cannot be written. The service preserves the already determined `PluginAuthError` and HTTP 401/403 semantics, emits only the fixed bounded denial reason to the audit-degradation monitor, and never converts an audit outage into authorization success or a generic availability response. This best-effort rule applies only after an event denial is known; destructive plugin operations still stop before applying an unaudited change.

HTTP status is derived from the typed plugin-auth error code, never from its message. Invalid or expired leases return `401`; a lease that belongs to another binding, a revoked binding, or a disabled principal returns `403`. A public plugin-auth operation that rejects an authentication-domain mismatch returns `403`, while presenting a portal bearer token to any plugin endpoint is rejected earlier at the HTTP boundary with `401`. Malformed public requests remain `400`, and a missing or invalid private operations identity remains `401`. Audit and monitoring failures do not replace the original typed denial or its status.

The principal pepper uses a pinned `v1` key ID in normal mode. Any key ID change requires explicit reviewed migration mode; a normal startup rejects the change so a disabled principal cannot be bypassed by deriving a new ID. Credential-pepper rotation invalidates all existing binding digests: revoke all bindings, rotate in a reviewed deployment, and require re-pairing.

## Operations review

Plugin operations use a separate private endpoint domain and a Google-signed OIDC bearer token with a plugin-ops-specific audience and an allowlist of service-account emails. Portal Firebase tokens, `x-goog-authenticated-user-email`, and body fields such as `approved_by` are rejected. Destructive operations use three endpoints: request, approve, and execute. The server generates an `opsrev_*` review ID and stores immutable action, target, parameters, requester, expiry, and status. A different verified service account must approve; execute atomically applies the stored action and consumes the review, so action/target substitution and replay fail.
